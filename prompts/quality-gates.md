# Quality Gates SSOT — 작업 완료 전 검증 강제

이 문서는 모든 영상 제작 / 톤 전환 / 일관성 보정 작업의 **"완료" 선언 전 필수 검증 게이트**다. PD는 이 게이트를 통과한 뒤에만 사용자에게 "완료" 보고할 수 있다.

근거: 메모리 [feedback_verify_before_complete.md] (2026-05-10 ep07-part3 작업 중 사용자 발견으로 추가됨).

## 게이트 0: 메모리 룰 원본 Read 강제 (룰 해석 단계)

메모리 룰을 적용하기 전에 **원본 .md 파일을 직접 Read**해서 정확한 정의 확인. 인덱스(MEMORY.md)의 한 줄 설명만 보고 추측으로 해석 금지.

**원칙**: "메모리 룰은 추측이 아니라 원본 정의에 근거한다."

**검증 절차**:
1. 룰 적용 직전 해당 메모 파일 Read
2. "How to apply" 섹션 정확히 확인
3. 다른 프로젝트의 실제 패턴과 cross-check (예: 다른 episodes의 oneline.txt 비율로 정의 검증)
4. 정의가 모호하면 사용자에게 확인 후 진행

**근거**: 2026-05-10 ep07-part3 작업 중 `feedback_script_oneline.md` 룰을 "한 줄 압축본"으로 잘못 해석. 실제 룰은 "줄바꿈을 공백으로 치환한 풀 텍스트"였음. 다른 43개 episodes는 99% 비율인데 ep07-part3만 14.7%로 작성. 사용자가 발견. PD가 다른 episodes와 cross-check 했더라면 즉시 발견 가능.

## 게이트 1: 에이전트 자가 보고 검증 (Trust but Verify)

에이전트가 "0건 잔존 / 통과 / 완료"를 자가 보고하더라도 PD가 무조건 직접 Grep으로 재검증한다.

**원칙**: "에이전트는 자기 누락을 모른다."

**검증 절차**:
1. 에이전트 작업 종료 직후
2. PD가 Grep 직접 실행 (자가 보고 신뢰 X)
3. 변형 패턴 최소 3차까지 검색
4. 0건 확인 전 다음 단계 진행 X

## 게이트 2: 변형 검색 패턴 강제 (좁은 패턴 = 누락 보장)

작업 종류별 검색 패턴 매트릭스:

### 톤 전환 작업 (보충편/답변 영상 등)
```
1차: 비판자|비판 댓글|한 분이|댓글창|그 댓글
2차: 비판 받|비판 들|비판 정면|비판 회수|비판자에|비판 핵심|비판 모드|비판 영상|비판 답변|비판 인정|비판 인용
3차: 비판\b (단독)
```

### YMYL 금융 정책 게이트 4 검증
```
1차: 사라|팔아라|확정|100%|무조건|뚫는다|절대
2차: 30%/70% 같은 종목 비중 권유 (예: "X 70 / Y 30", "OO 비중 N%로")
3차: 단정형 미래 보장 ("반드시 오른다", "100% 수익")
```

### 출처 인라인 강제 (게이트 4·9)
```
1차: 모든 % / 만원 / $ / 원 / 명 등 핵심 수치 카운트
2차: 핵심 수치 인근 5~10단어 내 출처 명시 여부 ("Total Real Returns", "한국경제 2025.01.20" 등)
```

### 13게이트 🔴 5개 1건 위반 = 채널 삭제 가능
```
3. 허위 조작 정보 — 가짜 사실 / 음모론 / 1차 출처 부재
5. 혐오 발언 — 보호 14종 비하 / 비인간화 / 폭력 선동

```

## 게이트 3: "완료" 선언 전 자가 체크리스트

PD는 사용자에게 "완료" 보고하기 전에 다음을 모두 ✅ 확인:

- [ ] **에이전트 자가 보고 외에** PD가 Grep 1회 이상 직접 실행
- [ ] **1차 검색 패턴** 0건 확인
- [ ] **2차 변형 패턴** 0건 확인 (1차 패턴이 좁을 가능성 대응)
- [ ] 영상 노출 산출물 (script.txt / script-oneline.txt / youtube.md / concept.md / hook-intro.md / prompts.json) 일관성 확인
- [ ] 작업 산출물 (draft.md / outline.md / patterns.md / review.md / _draft_partN.md / _refs/*) 일관성 확인 (영상 노출 X여도)
- [ ] 핵심 수치 보존 확인 (76%p, 601%p, 박대리 통장 등 — 작업별 다름)
- [ ] 출처 인라인 1회 이상 (Total Real Returns, PortfoliosLab 등)
- [ ] 메타데이터 정합성 (제목 = concept + youtube + prompts.json)
- [ ] 사용자 메모리 룰 12개 적용 (CTA 위치 / Q1 택일 / 시그니처 / 닭알 / 단정 X / 비중 X 등)

## 게이트 4: 잔존 발견 시 처리 절차

1차 / 2차 / 3차 패턴 검색에서 **1건이라도 잔존 발견 시**:

1. 발견 위치 명시 (파일·라인)
2. 매핑 표 작성 (잔존 표현 → 신규 표현)
3. PD 직접 Edit로 부분 수정 (에이전트 재호출 X — 다시 누락 위험)
4. 수정 후 같은 Grep 재실행
5. 0건 확인 후에만 다음 단계

**원칙**: 같은 누락 2회 이상 = 에이전트 신뢰 정지, PD 직접 마무리.

---

## 게이트 5: 사용자 발견 → PD 시스템 결함

사용자가 발견한 누락 = PD 시스템 결함 신호. 다음을 즉시 점검:

1. 검색 패턴이 좁았는가 (변형 추가 필요)
2. 에이전트 보고를 그대로 신뢰했는가 (PD 직접 검증 누락)
3. 검증 게이트가 작업 단계에 부재했는가
4. 메모리 룰이 부족했는가

→ 모두 ✅ 보강 후에만 다음 작업 진입.

## 자동화 레이어

이 SSOT는 다음 시스템과 연동:

- **메모리 룰**: `feedback_verify_before_complete.md` — PD 자율 검증 룰
- **PostToolUse Hook**: `.claude/hooks/verify-consistency.py` — 영상 노출 산출물 변경 후 자동 Grep 알림
- **슬래시 커맨드**: `/verify {project}` — 사용자 트리거 가능한 전수 검증
- **13게이트 SSOT**: `prompts/youtube-policy-gates.md` — STRATEGY/DRAFT/REVIEW 단계별 적용
